01 — Overview

What is OpenClaw?

OpenClaw is a self-hosted AI gateway that bridges your favorite chat apps — WhatsApp, Telegram, Discord, iMessage, and more — to AI coding agents and LLMs. You run a single Gateway process on your own machine (or a server), and it becomes the bridge between your messaging apps and an always-available AI assistant.

"At this point I don't even know what to call OpenClaw. It is something new. After a few weeks in with it, this is the first time I have felt like I am living in the future since the launch of ChatGPT." — @davemorin
🏠
Self-Hosted
Runs on your hardware, your rules. Your data stays yours — private by default.
💬
Multi-Channel
One Gateway serves WhatsApp, Telegram, Discord, and more simultaneously.
🧠
Agent-Native
Built for coding agents with tool use, sessions, memory, and multi-agent routing.
📖
Open Source
MIT licensed, community-driven. Fork it, hack it, make it yours.
ℹ️
Who is it for? Developers and power users who want a personal AI assistant they can message from anywhere — without giving up control of their data or relying on a hosted service.

Formerly known as Clawdbot and Moltbot. Built by Peter Steinberger and the community. Independent project, not affiliated with Anthropic.

02 — Features

Full Feature List

Channels

WhatsApp, Telegram, Discord, iMessage built-in. Mattermost, Matrix, Microsoft Teams, Nostr, and more via plugins. Group chat support with mention-based activation. DM safety with allowlists and pairing.

Agent Runtime

Embedded agent runtime with tool streaming. Multi-agent routing with isolated sessions per workspace or sender. Sessions: direct chats collapse into shared main; groups are isolated. Streaming and chunking for long responses.

Auth & Providers

35+ model providers (Anthropic, OpenAI, Google, and more). Subscription auth via OAuth. Custom and self-hosted provider support (vLLM, SGLang, Ollama, and any OpenAI-compatible or Anthropic-compatible endpoint).

Media

Images, audio, video, and documents in and out. Voice note transcription. Text-to-speech with multiple providers.

Apps & Interfaces

WebChat and browser Control UI. macOS menu bar companion app. iOS node with pairing, Canvas, camera, screen recording, location, and voice. Android node with pairing, chat, voice, Canvas, camera, and device commands.

Tools & Automation

Browser automation, exec, sandboxing. Web search (Brave, Perplexity, Gemini, Grok, Kimi, Firecrawl). Cron jobs and heartbeat scheduling. Skills, plugins, and workflow pipelines (Lobster).

03 — Installation

Installing OpenClaw

System Requirements: Node 24 (recommended) or Node 22.16+. macOS, Linux, or Windows (native or WSL2). WSL2 is more stable than native Windows and recommended for the full experience.

✅
Fastest path — The installer script handles everything: detects your OS, installs Node if needed, installs OpenClaw, and launches onboarding.
1

Install via one-liner (macOS / Linux / WSL2)

curl -fsSL https://openclaw.ai/install.sh | bash

Windows (PowerShell):

iwr -useb https://openclaw.ai/install.ps1 | iex
2

Run onboarding

openclaw onboard --install-daemon

The wizard walks you through choosing a model provider, setting an API key, and configuring the Gateway. Takes about 2 minutes.

3

Verify the Gateway is running

openclaw gateway status

You should see the Gateway listening on port 18789.

4

Open the dashboard

openclaw dashboard

This opens the Control UI in your browser. Type a message and you should get an AI reply!

Alternative Install Methods

npm / pnpm

npm install -g openclaw@latest
openclaw onboard --install-daemon
# pnpm
pnpm add -g openclaw@latest
pnpm approve-builds -g
openclaw onboard --install-daemon

From Source

git clone https://github.com/openclaw/openclaw.git
cd openclaw
pnpm install && pnpm ui:build && pnpm build
pnpm link --global
openclaw onboard --install-daemon

Install from GitHub main

npm install -g github:openclaw/openclaw#main

Verify Your Install

openclaw --version      # confirm the CLI is available
openclaw doctor         # check for config issues
openclaw gateway status # verify the Gateway is running
⚠️
openclaw not found? Check that $(npm prefix -g)/bin is in your $PATH. Add export PATH="$(npm prefix -g)/bin:$PATH" to your ~/.zshrc or ~/.bashrc and open a new terminal.

06 — Core Concepts

Gateway Architecture

The Gateway is the single, always-on process that controls everything — messaging surfaces, routing, control plane, and channel connections.

Key Components

🌐
Gateway (Daemon)
Maintains provider connections. Exposes a typed WebSocket API. Emits events: agent, chat, presence, health, heartbeat, cron.
🖥️
Clients
macOS app, CLI, web admin. One WebSocket connection per client. Send requests, subscribe to events.
📱
Nodes
macOS / iOS / Android / headless. Connect with role:node. Expose commands like canvas.*, camera.*, location.get.
💻
WebChat
Static UI using the Gateway WebSocket API. Available at http://127.0.0.1:18789/ by default.

Runtime Model

One always-on process for routing, control plane, and channel connections. Single multiplexed port for: WebSocket control/RPC, HTTP APIs (OpenAI-compatible, Responses, tools invoke), Control UI and hooks. Default bind mode: loopback. Auth is required by default.

Port & Bind Settings

SettingResolution Order
Gateway port--port → OPENCLAW_GATEWAY_PORT → gateway.port → 18789
Bind modeCLI/override → gateway.bind → loopback

Hot Reload Modes

ModeBehavior
offNo config reload
hotApply only hot-safe changes
restartRestart on reload-required changes
hybrid (default)Hot-apply when safe, restart when required

Operator Commands

openclaw gateway status       # check status
openclaw gateway status --deep # deep diagnostics
openclaw gateway status --json # JSON output
openclaw gateway install       # install as service
openclaw gateway restart       # restart the gateway
openclaw gateway stop          # stop the gateway
openclaw secrets reload        # reload secrets
openclaw logs --follow         # tail logs
openclaw doctor                # audit and repair

Remote Access

Preferred: Tailscale or VPN. Fallback: SSH tunnel:

ssh -N -L 18789:127.0.0.1:18789 user@host

08 — Memory & Sessions

Memory & Sessions

How Memory Works

OpenClaw memory is plain Markdown in the agent workspace. The files are the source of truth; the model only "remembers" what gets written to disk.

📅
Daily Log
memory/YYYY-MM-DD.md — Append-only daily log. Reads today + yesterday at session start.
🧠
Long-term Memory
MEMORY.md — Curated long-term memory at the workspace root. Decisions, preferences, durable facts.

Memory Tools

Two agent-facing tools are available:

ToolPurpose
memory_searchSemantic recall over indexed snippets
memory_getTargeted read of a specific Markdown file or line range

When to Write Memory

Decisions, preferences, and durable facts → MEMORY.md. Day-to-day notes → memory/YYYY-MM-DD.md. If someone says "remember this," write it down immediately. If you want something to stick, ask the bot to write it into memory.

Vector Memory Search

OpenClaw can build a small vector index over memory files for semantic queries. Supports hybrid search (BM25 + vector). Multiple embedding providers: OpenAI, Gemini, Voyage, Mistral, Ollama, and local GGUF models. See Memory Config Reference →

Session Management

Sessions track the conversation history for each agent. Direct chats collapse into a shared main session. Groups get isolated sessions. See: Session Management → | Compaction →

Automatic Memory Flush

When a session nears auto-compaction, OpenClaw triggers a silent turn to write durable memory before context is compacted. Configurable via agents.defaults.compaction.memoryFlush.

11 — Skills

Skills System

Skills teach the agent when and how to use tools. A skill is a Markdown file (SKILL.md) injected into the system prompt with YAML frontmatter and instructions.

Three Layers

1

Tools — What the agent calls

A tool is a typed function the agent can invoke (e.g. exec, browser, web_search, message).

2

Skills — Teach the agent when and how

A skill is a Markdown file injected into the system prompt. Skills give context, constraints, and step-by-step guidance for using tools effectively.

3

Plugins — Package everything together

A plugin registers any combination: channels, model providers, tools, skills, speech, image generation, and more.

Skill Locations & Precedence

Skills are loaded from three places (highest wins):

  1. <workspace>/skills — highest precedence (per-agent)
  2. ~/.openclaw/skills — managed/local skills (shared across agents)
  3. Bundled skills — shipped with the install (lowest precedence)

SKILL.md Format

---
name: image-lab
description: Generate or edit images via a provider-backed image workflow
metadata:
  {
    "openclaw": {
      "emoji": "🖼️",
      "requires": { "bins": ["uv"], "env": ["GEMINI_API_KEY"] },
      "primaryEnv": "GEMINI_API_KEY",
    },
  }
---

13 — Automation

Automation & Cron Jobs

Cron is the Gateway's built-in scheduler. It persists jobs, wakes the agent at the right time, and can optionally deliver output back to a chat. Jobs persist at ~/.openclaw/cron/jobs.json.

Quick Start

# One-shot reminder
openclaw cron add \
  --name "Reminder" \
  --at "2026-02-01T16:00:00Z" \
  --session main \
  --system-event "Reminder: check the cron docs draft" \
  --wake now \
  --delete-after-run

# Recurring isolated job with Slack delivery
openclaw cron add \
  --name "Morning brief" \
  --cron "0 7 * * *" \
  --tz "America/Los_Angeles" \
  --session isolated \
  --message "Summarize overnight updates." \
  --announce \
  --channel slack \
  --to "channel:C1234567890"

Schedule Kinds

KindUse forExample
atOne-shot timestamp--at "2026-01-12T18:00:00Z"
everyFixed interval (ms)--every 300000
cron5/6-field cron expression--cron "0 7 * * *"

Session Targets

TargetBehavior
mainRun during next heartbeat with main context
isolatedDedicated agent turn in cron:<jobId>
currentBind to the session where cron was created
session:custom-idPersistent named session (context preserved across runs)

Delivery Modes

ModeBehavior
announceDeliver output to target channel, brief summary to main session
webhookPOST finished event payload to a URL
noneInternal only, no delivery

Retry Policy

Transient errors (retried): Rate limit, provider overload, network errors, 5xx server errors.

Permanent errors (disable immediately): Auth failures, config/validation errors.

17 — Configuration

Configuration Reference

Config lives at ~/.openclaw/openclaw.json. If you do nothing, OpenClaw uses sensible defaults. Example config:

{
  // Channel config
  channels: {
    whatsapp: {
      allowFrom: ["+15555550123"],
      groups: { "*": { requireMention: true } },
    },
  },

  // Default model
  agents: {
    defaults: {
      model: { primary: "anthropic/claude-opus-4-6" },
      workspace: "~/.openclaw/workspace",
    },
  },

  // Tool restrictions
  tools: {
    profile: "coding",
    allow: ["group:fs", "browser", "web_search"],
    deny: ["exec"],
  },

  // Cron settings
  cron: {
    enabled: true,
    sessionRetention: "24h",
  },

  // Gateway settings
  gateway: {
    port: 18789,
    bind: "loopback",
    reload: { mode: "hybrid" },
    auth: { token: "your-secret-token" },
  },

  // Skills
  skills: {
    entries: {
      "my-skill": { enabled: true },
    },
  },
}

18 — Security

Security & Sandboxing

⚠️
Treat third-party skills as untrusted code. Read them before enabling. Prefer sandboxed runs for untrusted inputs and risky tools.

Pairing & Trust

All WebSocket clients include a device identity on connect. New device IDs require pairing approval. The Gateway issues a device token for subsequent connects. Local connects (loopback) can be auto-approved.

Allowlists

Control who can message your agent with channels.*.allowFrom per channel. Use groups.*.requireMention to require @mention in group chats.

Sandboxing

Run the agent in a sandboxed Docker container for untrusted inputs. Configure via agents.defaults.sandbox.docker. See Sandboxing →

Formal Verification & Threat Model

OpenClaw publishes a formal threat model using MITRE ATLAS. See: Formal Verification → | Threat Model →
